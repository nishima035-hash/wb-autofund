# WB AutoFund — локальный MVP

Рабочий прототип автопополнения рекламных кампаний Wildberries. Проверяет остаток, максимальный ДРР, минимальный CTR и дневной лимит. Все решения записываются в журнал SQLite; токен хранится в AES-256-GCM зашифрованном виде.

## Запуск

Требуется Node.js 22.5 или новее (используется встроенный SQLite).

1. Скопируйте `.env.example` в `.env`.
2. Сгенерируйте уникальный `APP_ENCRYPTION_KEY` командой из комментария в `.env.example` и вставьте его в файл.
3. Запустите `npm start`.
4. Откройте http://localhost:4173.

При первом запуске включён демо-режим и созданы три тестовые кампании. Настройте правило, включите его и нажмите «Проверить сейчас».

## Подключение WB

Создайте токен с категориями «Продвижение» и «Статистика» в кабинете WB, откройте вкладку «Подключение WB», вставьте токен, выключите демо-режим и сохраните. Кнопка «Обновить из WB» загружает кампании, бюджеты и статистику. Для каждого добавленного юрлица синхронизация и история хранятся отдельно.

По умолчанию настоящее пополнение заблокировано двумя независимыми условиями: демо-режим должен быть выключен и `WB_LIVE_DEPOSITS=true` в `.env`. Сначала проверьте синхронизацию и журнал. Источник списания задаётся в правиле: счёт (0), баланс (1), бонусы (3).

В разделе «Показы по часам» показы и клики рассчитываются как разница накопительных снимков API продвижения. Рекомендуемый интервал обновления — 5 минут. Если между снимками большой разрыв или счётчик WB сбросился, ячейка помечается знаком `!`, а недостоверное значение не маскируется под точное.

Заказы берутся из API «Статистика» и раскладываются по фактическому московскому времени заказа. WB обычно обновляет этот источник с задержкой, поэтому свежий заказ может появиться позднее, но после синхронизации попадёт в свой исходный час. Привязка заказа к рекламной кампании выполняется по артикулу `nmId`; если один товар одновременно находится в нескольких кампаниях, результат помечается знаком `!`, поскольку API заказов не сообщает ID рекламной кампании.

## Защита

- уникальный idempotency key на кампанию и минуту;
- блокировка кампании на время операции;
- дневной лимит считается по успешным пополнениям за календарные сутки Москвы (`Europe/Moscow`, UTC+3), независимо от часового пояса сервера;
- токен не возвращается в браузер и не попадает в журнал;
- боевые операции выключены по умолчанию;
- SQLite работает в WAL-режиме.

## Границы MVP

Это однопользовательский локальный прототип. Для публичного SaaS нужны авторизация, PostgreSQL, очередь задач, централизованное управление секретами, повтор запросов с backoff/учётом rate limit, мониторинг, резервные копии и сверка idempotency с историей операций WB.
